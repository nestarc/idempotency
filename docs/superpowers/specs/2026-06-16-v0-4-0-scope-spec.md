# v0.4.0 Scope Specification

**Date:** 2026-06-16
**Package:** `@nestarc/idempotency`
**Current version:** `0.3.0`
**Target version:** `0.4.0`
**Status:** Draft for user review

## Goal

Define the product and technical scope for `@nestarc/idempotency` v0.4.0.

v0.4.0 should move the package from a correctness-focused HTTP idempotency
module into a production-ready NestJS retry-safety layer. The release should
focus on the highest-confidence gaps found during codebase, ecosystem,
competitor, and community research:

1. Requests stuck in `PROCESSING` after crashes or long failures.
2. Webhook and business-command idempotency that cannot rely only on the
   `Idempotency-Key` header.
3. Operational visibility into created, replayed, conflicted, mismatched, and
   stale requests.

## Positioning

`@nestarc/idempotency` should be positioned as a **NestJS data-safety
infrastructure module**. It protects inbound state-changing HTTP requests in
the same broader safety layer where `@nestarc/soft-delete`, `@nestarc/audit-log`,
`@nestarc/outbox`, `@nestarc/webhook`, and `@nestarc/tenancy` operate.

It should not try to become a general queue deduplication engine in v0.4.0.
The package's strongest near-term position is:

> Nest-first, Redis/Postgres-capable, production-grade idempotency for mutation
> endpoints, payments, orders, and webhook receivers.

## Evidence Summary

### Codebase facts

- `IdempotencyInterceptor` owns the current request state machine:
  missing key, mismatch, processing conflict, create, complete, replay, and
  cleanup.
- Current storage adapters are `MemoryStorage`, `RedisStorage`, and
  `PostgresStorage`.
- Current storage contract supports atomic create and token-CAS complete/delete.
- Current public options include `ttl`, `headerName`, `fingerprint`, `scope`,
  `replayHeaders`, and `isGlobal`.
- Current gaps: no processing lease separate from replay TTL, no custom key
  resolver, no custom fingerprint resolver, no observability hook, no
  business-error caching policy, and no Swagger/OpenAPI helper.

### External and community facts

- Payment APIs such as Stripe, Square, Adyen, and PayPal document idempotency as
  a retry-safety mechanism for network timeouts and duplicate submissions.
- AWS guidance emphasizes caller-provided request identifiers and the difficulty
  of keeping idempotency records and business mutations in a safe boundary.
- Express and Hono competitors already expose some combination of processing
  timeout, status headers, custom fingerprints, hooks, or error replay.
- Community issues show real production pain around long-lived `PROCESSING`
  records becoming poison keys after an interrupted request.

## Scope Principles

1. Preserve v0.3.0 default behavior unless a user opts into a new policy.
2. Keep the storage contract small and adapter-neutral.
3. Prefer route-level and module-level options over global-only behavior.
4. Do not claim full exactly-once execution across user business transactions.
5. Describe IETF behavior as a draft-compatible profile, not an RFC guarantee.
6. Add tests before implementation for every new state-machine branch.
7. Favor features with at least two signals among code gap, user pain,
   competitor behavior, and community demand.

## Must-Have Scope

The following items should be included in v0.4.0 unless implementation reveals a
blocking design flaw.

## Must 1: Processing Lease / `processingTtl`

### Problem

Today, a request that acquires a key creates a `PROCESSING` record using the same
TTL policy as completed replay records. If the server crashes, the handler hangs,
or completion never writes, retries can receive `409 Conflict` until the full
record TTL expires.

That behavior is safe but operationally painful for payments, orders, imports,
and webhook receivers.

### User value

- Reduces manual key deletion during incidents.
- Lets services choose a shorter in-flight lock window than the replay window.
- Makes long replay TTLs practical without making processing failures block
  clients for the same duration.

### Proposed API

Add a processing-specific TTL while keeping the current default:

```ts
export interface IdempotencyOptions {
  ttl: number;
  processingTtl?: number;
}

export interface IdempotentOptions {
  ttl?: number;
  processingTtl?: number;
}
```

Behavior:

- If `processingTtl` is omitted, use `ttl` for `PROCESSING` records. This keeps
  v0.3.0 behavior.
- `storage.create()` should receive `processingTtl`.
- `storage.complete()` should continue to receive `ttl` for completed replay
  lifetime.
- `processingTtl` must be a positive integer in seconds.
- Documentation must warn that too-short processing leases can allow duplicate
  execution if the first request is still running after the lease expires.

### Acceptance criteria

- Existing tests pass with no options changed.
- A route configured with `ttl: 86400` and `processingTtl: 30` creates a
  processing record that expires after 30 seconds and a completed record that
  replays for 86400 seconds.
- A late `complete()` from an expired processing token must return stale and
  must not overwrite a newer record.
- Memory, Redis, and Postgres adapters behave consistently.
- README documents how to choose `processingTtl` using endpoint p99 latency plus
  buffer.

## Must 2: Custom Key Resolver

### Problem

The current package assumes the idempotency key comes from a request header.
That works for client-driven APIs but is insufficient for webhook receivers and
some business commands where the stable key is a provider event id, order id,
cart id, tenant id, or command id.

### User value

- Enables Stripe/GitHub/Shopify-style webhook receiver deduplication.
- Allows application-level command ids without forcing clients to send
  `Idempotency-Key`.
- Improves multi-tenant key isolation when paired with custom scope.

### Proposed API

```ts
import type { ExecutionContext } from '@nestjs/common';

export type IdempotencyKeyResolver = (
  context: ExecutionContext,
) => string | undefined | Promise<string | undefined>;

export interface IdempotencyOptions {
  keyResolver?: IdempotencyKeyResolver;
  maxKeyLength?: number;
}

export interface IdempotentOptions {
  keyResolver?: IdempotencyKeyResolver;
  maxKeyLength?: number;
}
```

Resolver precedence:

1. Route-level `@Idempotent({ keyResolver })`.
2. Module-level `IdempotencyModule.forRoot({ keyResolver })`.
3. Header lookup using `headerName`.

If no key is resolved:

- `required: true` returns the existing missing-key `400`.
- `required: false` passes through without idempotency protection.

`maxKeyLength` should default to 255 characters, matching common API-provider
limits. If the key exceeds the limit, return `400`.

### Acceptance criteria

- Header behavior remains unchanged when no resolver is configured.
- Route-level resolver overrides module-level resolver.
- Async resolver is supported.
- Resolver returning `undefined` respects `required`.
- Resolver errors fail the request and do not create storage records.
- Tests cover webhook-style key extraction from body and headers.

### Related files

- `src/interfaces/idempotency-options.interface.ts`
- `src/idempotency.decorator.ts`
- `src/idempotency.interceptor.ts`
- `test/idempotency.interceptor.spec.ts`

## Must 3: Custom Fingerprint Resolver

### Problem

The current fingerprint uses stable JSON over the request body. This is a good
default, but real APIs often need a semantic fingerprint:

- ignore volatile fields such as timestamps or client metadata;
- include tenant, user, provider, or route context;
- fingerprint a normalized DTO instead of raw body order;
- use webhook provider event ids or object ids.

### User value

- Prevents false mismatches caused by harmless payload differences.
- Prevents false matches when two requests share a body but differ in tenant or
  semantic target.
- Supports payment and webhook recipes without custom forks.

### Proposed API

Extend the existing `fingerprint` option without breaking the boolean form:

```ts
import type { ExecutionContext } from '@nestjs/common';

export interface IdempotencyFingerprintInput {
  context: ExecutionContext;
  key: string;
  scope: string;
  body: unknown;
  defaultFingerprint: () => string | undefined;
}

export type IdempotencyFingerprintResolver = (
  input: IdempotencyFingerprintInput,
) => string | undefined | Promise<string | undefined>;

export interface IdempotencyOptions {
  fingerprint?: boolean | IdempotencyFingerprintResolver;
}

export interface IdempotentOptions {
  fingerprint?: boolean | IdempotencyFingerprintResolver;
}
```

Behavior:

- `fingerprint: true` keeps current stable JSON behavior.
- `fingerprint: false` keeps current no-fingerprint behavior.
- Resolver result is stored as the fingerprint value.
- Resolver returning `undefined` behaves like no fingerprint for that request.
- Resolver throw fails the request and must not create a storage record.

### Acceptance criteria

- Existing fingerprint tests continue to pass.
- Custom resolver can ignore a volatile field and replay equivalent requests.
- Custom resolver can include tenant context and reject same key/body across
  different tenants.
- Async resolver is supported.
- Same key plus different custom fingerprint still returns `422`.

## Must 4: Observability Events and Status Headers

### Problem

The package currently protects the request but does not give operators a clean
way to measure behavior. Teams need to know whether requests were created,
replayed, rejected as in-flight, rejected as mismatched, bypassed, or failed due
to storage errors.

### User value

- Easier incident debugging.
- Better dashboards for replay rate, conflict rate, mismatch rate, and storage
  reliability.
- Better client debugging through explicit response headers.

### Proposed API

```ts
export type IdempotencyOutcome =
  | 'created'
  | 'replayed'
  | 'conflict'
  | 'mismatch'
  | 'bypassed'
  | 'stale'
  | 'complete_error'
  | 'storage_error';

export interface IdempotencyEvent {
  outcome: IdempotencyOutcome;
  keyHash: string;
  scope: string;
  method?: string;
  path?: string;
  statusCode?: number;
  durationMs?: number;
  error?: unknown;
}

export interface IdempotencyObservabilityOptions {
  onEvent?: (event: IdempotencyEvent) => void | Promise<void>;
  exposeStatusHeaders?: boolean;
}

export interface IdempotencyOptions {
  observability?: IdempotencyObservabilityOptions;
}
```

Header behavior when `exposeStatusHeaders !== false`:

- First successful execution: `Idempotency-Status: created`
- Replay: `Idempotency-Status: replayed` and `Idempotency-Replayed: true`
- Processing conflict: `Idempotency-Status: conflict`
- Fingerprint mismatch: `Idempotency-Status: mismatch`

Raw keys must not be emitted in events. Use a SHA-256 hash or another stable
redacted representation for `keyHash`.

### Acceptance criteria

- Every state-machine outcome emits at most one event.
- `onEvent` failures are swallowed and logged; they must not change request
  outcome.
- Existing replay header behavior remains compatible.
- Status headers are present for created, replayed, conflict, and mismatch
  paths when enabled.
- Tests cover event payload redaction.

- `src/utils/response-headers.ts`
- `test/idempotency.interceptor.spec.ts`
- `test/e2e/idempotency.e2e-spec.ts`

## Must 5: Draft-Compatible Compliance and Documentation Cleanup

### Problem

The package currently describes itself as IETF-draft-compliant, while the
referenced `draft-ietf-httpapi-idempotency-key-header-07` is an expired
Internet-Draft. The behavior can still be draft-compatible, but the wording
should avoid implying final RFC status.

### User value

- More accurate trust posture.
- Less confusion about what is guaranteed by the package.
- Clearer upgrade guidance from v0.3.0 to v0.4.0.

### Required documentation updates

- Use "IETF draft-07-compatible" or "draft-aligned" instead of "RFC compliant".
- Document the exact behavior matrix:
  - missing key;
  - first execution;
  - replay;
  - in-flight conflict;
  - same key plus different fingerprint;
  - binary/bypass response;
  - storage complete failure;
  - processing lease expiry.
- Explain that the package does not guarantee full exactly-once execution across
  user business database transactions.
- Add examples for:
  - payment/order endpoint;
  - webhook receiver with custom key resolver;
  - tenant-aware fingerprint or scope;
  - metrics hook.

### Acceptance criteria

- README and CHANGELOG use draft-compatible wording.
- `IdempotencyScope` comments match actual path-based endpoint scoping.
- `docs/handover.md` no longer contradicts v0.3.0 header replay behavior.
- v0.4.0 examples can be copied into a NestJS app without relying on hidden
  internal APIs.

### Related files

- `README.md`
- `CHANGELOG.md`
- `src/interfaces/idempotency-options.interface.ts`
- `docs/handover.md`

## Should-Have Scope

The following items are valuable and should be included if the must-have scope
is stable early enough. They should not delay the release if they expand risk.

## Should 1: Business Error Caching Policy

### Problem

Today, if a handler throws, the package deletes the processing record. This is
safe for unknown server failures, but some domain errors are deterministic and
should be replayed for the same idempotency key.

Example: a payment request fails with a deterministic `402` or an order command
fails with a deterministic `409` domain conflict. Re-running the handler may
repeat validation, provider calls, logging, or other side effects.

### Proposed API

```ts
export type CacheErrorsOption =
  | false
  | '4xx'
  | '5xx'
  | 'all'
  | number[]
  | ((statusCode: number, error: unknown) => boolean);

export interface IdempotencyOptions {
  cacheErrors?: CacheErrorsOption;
}

export interface IdempotentOptions {
  cacheErrors?: CacheErrorsOption;
}
```

Default must be `false`.

### Acceptance criteria

- Default error behavior remains unchanged.
- Configured `4xx` errors are completed and replayed.
- `5xx` errors are not cached unless explicitly configured.
- Cached error body must come from `HttpException.getResponse()` when available.
- Tests prove duplicate deterministic errors run the handler only once.

### Risk

Caching server errors can hide transient recovery. Keep this opt-in and document
safe defaults.

## Should 2: Swagger/OpenAPI Helper

### Problem

API consumers need to know which endpoints require `Idempotency-Key` and what
`400`, `409`, and `422` mean.

### Proposed API

Provide an optional helper that does not make `@nestjs/swagger` a hard runtime
dependency for core users:

```ts
export function ApiIdempotencyKey(options?: {
  required?: boolean;
  headerName?: string;
}): MethodDecorator & ClassDecorator;
```

Possible implementation approaches:

- separate `src/swagger` entrypoint;
- lazy `require('@nestjs/swagger')`;
- optional peer dependency documented in README.

### Acceptance criteria

- Core package import works without `@nestjs/swagger` installed.
- Helper adds the idempotency header and standard error responses.
- Tests snapshot generated metadata or decorator effects.

## Should 3: In-Flight Wait Strategy

### Problem

Current in-flight duplicates get `409`. That is draft-compatible and safe, but
some mobile and payment clients prefer waiting briefly for the first request to
complete and then receiving replay.

### Proposed API

```ts
export interface IdempotencyInFlightOptions {
  strategy?: 'conflict' | 'wait';
  waitTimeoutMs?: number;
  pollMs?: number;
}

export interface IdempotencyOptions {
  inFlight?: IdempotencyInFlightOptions;
}
```

Default strategy must remain `conflict`.

### Acceptance criteria

- Default behavior remains `409`.
- `wait` polls storage until completed or timeout.
- Timeout returns `409` with `Retry-After` when configured.
- Polling has sane minimums to prevent tight loops.

### Risk

This can increase request latency and storage load. It should be opt-in and
documented as a client UX feature, not a correctness requirement.

## Should 4: Storage Contract Testing Export

### Problem

The repo has a shared storage contract, but external adapter authors cannot
import it from the npm package.

### Proposed API

Expose a testing-only subpath:

```ts
import { describeIdempotencyStorageContract } from '@nestarc/idempotency/testing';
```

### Acceptance criteria

- npm package includes the testing entrypoint.
- The helper works with a fake external storage adapter.
- The helper does not pull test-only dependencies into the core runtime import.

## Should 5: Ecosystem Recipes

Add documentation recipes rather than hard dependencies:

- `@nestarc/tenancy` tenant-aware scope.
- `@nestarc/api-keys` application-aware scope.
- `@nestarc/outbox` boundary explanation.
- inbound webhook receiver dedupe.
- payment/order endpoint recipe.

Acceptance criteria:

- Recipes do not require installing other nestarc packages to understand the
  idempotency API.
- Examples avoid claiming transactional exactly-once semantics.

## Should 6: Small API Hygiene

If it does not expand scope, fix these small inconsistencies:

- Export `PostgresStorage` from `src/storage/index.ts`.
- Ensure package metadata and README use the same package positioning.
- Confirm `@nestjs/platform-fastify` is installed in local dev dependencies so
  `npm run test:e2e` is reproducible.

## Not in v0.4.0

The following items are intentionally excluded from the stable v0.4.0 scope.

## Not Now 1: Stable `@TransactionalIdempotent`

Transactional integration is strategically important, but it is too risky to
ship as a broad stable decorator in v0.4.0.

Reasons:

- It crosses package boundaries into user business database transactions.
- Prisma, TypeORM, Drizzle, raw `pg`, and custom transaction managers need
  different integration points.
- A shallow wrapper could create a stronger exactly-once impression than the
  package actually guarantees.

Allowed in v0.4.0:

- design note;
- experimental Postgres-only primitive;
- example showing the transaction boundary and known failure modes.

Stable transactional support should be reconsidered for v0.5.0 after crash-window
tests and ORM-specific integration designs exist.

## Not Now 2: `IdempotencyService.run()` for Jobs and Consumers

HTTP-outside execution is useful for queues and workers, but it would shift the
package from HTTP mutation safety to general distributed execution.

Reconsider after v0.4.0 if:

- `@nestarc/jobs` or `@nestarc/webhook` needs a shared dedupe primitive;
- storage contract can support non-HTTP response payloads cleanly;
- a separate service-level API can be designed without weakening interceptor
  semantics.

## Not Now 3: New Built-In Storage Adapters

Do not add DynamoDB, MongoDB, MySQL, Cloudflare KV, or Durable Objects adapters
in v0.4.0.

First stabilize:

- processing lease behavior;
- custom key/fingerprint API;
- public storage contract testing.

## Not Now 4: Dual ESM/CJS Build

Dual package output is useful but not tied to the highest-priority user pain.
It should not compete with runtime correctness and observability work in v0.4.0.

## Release Success Criteria

v0.4.0 is successful when:

- existing 0.3.0 users can upgrade without changing default behavior;
- processing lease can prevent long poison-key incidents when explicitly
  configured;
- webhook and business-command dedupe can be implemented using public APIs;
- every idempotency outcome can be observed through events and headers;
- docs accurately describe draft-compatible behavior and limits;
- unit, e2e, regression, Redis, and Postgres storage contract tests pass.

## Suggested Implementation Order

1. Add failing tests for `processingTtl`.
2. Implement `processingTtl` through interceptor and storage adapters.
3. Add failing tests for `keyResolver`.
4. Implement key resolver precedence and validation.
5. Add failing tests for custom fingerprint resolver.
6. Implement resolver-based fingerprinting.
7. Add event and status-header tests.
8. Implement observability hook and redacted event payload.
9. Update README, CHANGELOG, and stale comments.
10. Add should-have items only after must-have tests are stable.
